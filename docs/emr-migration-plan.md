# 電子カルテ切替 移行作業洗い出し

## 前提（ヒアリング結果）

- 切替後もEyeData（検査・手術記録・サマリー・予約機能）は継続使用する。改修対象は「現行電子カルテ（InnoKarte）との連携部分」のみで、EyeData自体の機能・DB構造は維持する想定。
- EYE_KENSA / EYE_OPE / EYE_OPE_RECORD / EYE_SUMMARY が保持するPATIENT_IDは、新電子カルテでも同じ患者IDがそのまま使われる想定。
- データ移行はEyeDataの既存CSVエクスポート機能（`FormExport.cs`）を使い、取込先（新電子カルテへの取込・マッピング・検証）も自院側で行う。
- 新電子カルテの製品は未定。そのため以下は現時点で確定できず、製品選定後に個別確認・改修が必要な項目として記載する。

## 現状のInnoKarte依存箇所（要改修見込み）

- `FormPat.cs` の `InnoButton_Click` → `InnoProgram.KarteShow(this.Pat.Id)`（患者画面から現行カルテを起動するボタン）。`InnoProgram.Exists` が false の場合はボタン自体を無効化する分岐が既にある。
- ログインユーザー取得元 `C:\innokarte\pat.csv`（`LoginUser.ReadPatCSV`、50カラムShift-JIS、患者ID/氏名/ログインユーザーIDを含む）。
- Oracle DBリンク `INNO.WORLD`（`M_PATIENT@INNO.WORLD`）。検索系は2026-07-17の改修でDBリンク結合を廃止し `PatBase.GetDict` 経由の2段階取得に変更済みだが、`EyeOpe.GetListByKindDates` 系など未対応で残っている箇所がある（[[search-two-stage-design]]参照）。
- `InnoUketsukeLib.dll`（`LibSettings.Init` / `Staff.Verify` から呼ばれるランタイムオプション依存。無い場合はパスワード照合をスキップしてID照合のみにフォールバック）。
- `MedicalLibrary_Settings.xml` / `EyeData.xml`（`EyeData.xml`）内のInno関連設定値（パスやDB接続情報など、内容は未調査）。

> 上記は本リポジトリ（EyeData）内で確認できた範囲。実際の接続処理・認証ロジックの大部分は `MedicalLibrary`（隣接プロジェクト）側にあるため、改修時はMedicalLibrary側の調査・改修も必要になる可能性が高い。

## 作業リスト

### 1. 新電子カルテ選定・要件確認（製品確定前に必須）

- [ ] 新電子カルテ製品の選定（未定）
- [ ] 新電子カルテがCSVインポート機能を持つか、対応フォーマット（列構成・区切り文字・文字コード・改行の扱い）を確認
- [ ] 新電子カルテの患者ID体系を確認し、現行PATIENT_IDをそのまま使えるか、変換が必要かを確定
  - 変換が必要な場合、旧ID⇔新IDの対応表作成・EYE_*テーブルのPATIENT_ID一括変換が追加作業として発生する
- [ ] 新電子カルテが外部システムからの患者情報参照・カルテ起動連携（現行の`InnoProgram.KarteShow`相当）をサポートするか、APIやプロトコルの有無を確認
- [ ] 新電子カルテのログイン認証方式を確認（現行はID+パスワードで`InnoUketsukeLib`に照合、DLL不在時はIDのみでフォールバック）
- [ ] 新電子カルテ側のデータベース種別・接続方式（Oracle/DBリンク可否など）を確認
- [ ] インポート作業の実施者・タイムスケジュール（誰がいつ取り込むか）を確認

### 2. EyeData側の連携改修（現行Inno依存の置き換え）

- [ ] `FormPat.cs` のInnoKarte起動ボタン（`InnoButton`）を新電子カルテ向けの起動方式に置き換え、または新電子カルテが同等機能を持たない場合はボタンの扱い（無効化/削除）を決定
- [ ] ログインユーザー取得元（`Pat.csv`読込）を新電子カルテのユーザー情報取得方式に置き換え
- [ ] DBリンク`INNO.WORLD`（`M_PATIENT@INNO.WORLD`）参照箇所の洗い出しと置き換え（未対応箇所を含む全数調査が必要）
- [ ] `InnoUketsukeLib.dll`依存箇所（`MedicalLibrary`側）の扱いを決定（新電子カルテ用の同等ライブラリに置き換えるか、認証方式自体を変更するか）
- [ ] 設定ファイル（`MedicalLibrary_Settings.xml`, `EyeData.xml`）のInno関連設定を新電子カルテ向けに更新
- [ ] MedicalLibrary側の関連クラス（`LoginUser` / `PatBase` / `LibSettings`等）に新電子カルテ固有の改修が必要か調査（本リポジトリ外のため別途調査要）
- [ ] 改修後、DLL/exeを一式そろえて配置（過去の教訓：シグネチャ変更時はMedicalLibrary.dllとEyeData.exeを必ずセットで配置。[[build-dependency-placement]]参照）

### 3. データ移行（エクスポート→新電子カルテ取込）

- [ ] 移行対象データの確定：患者（Pat）／検査（Kensa=EYE_KENSA）／手術記録（OpeRecord=EYE_OPE_RECORD+EYE_OPE）／手術予定（Rsv=EYE_OPE）／サマリー（Summary=EYE_SUMMARY）
- [ ] 各CSVの列構成と新電子カルテのインポート仕様との突き合わせ（列名・順序・必須項目の過不足）
- [ ] 文字コードの決定（現行はUTF-8 BOM付き/Shift-JISを選択可能。新電子カルテ側の対応コードに合わせる）
- [ ] 改行を含むセル（`CONT`列等）の`<CR+LF>`トークン変換の扱いを新電子カルテ取込側で正しく復元できるか確認、または取込前に変換する処理の要否を確認
- [ ] `SumItem1`〜`4` / `OpeTabItem`（`EyeDict.EyeSet`）由来の項目コード・ラベルが新電子カルテ側のコード体系と対応するかを確認し、必要であれば項目マッピング表を作成
- [ ] 患者マスタ非存在ID（`SavePat`で空欄出力される想定外PATIENT_ID）の扱いを確認・是正
- [ ] PATIENT_IDが新電子カルテで変わる場合、EYE_*系CSV内のPATIENT_IDを新IDへ変換する処理を追加作成
- [ ] 大量データ export（PAGE_SIZE=5000のページング）の所要時間・DB負荷を事前に見積り、業務時間外実施の要否を判断
- [ ] エクスポート結果の`.manifest.txt`（件数・出力日時）を用いた件数検証を移行手順に組み込む
- [ ] 取込リハーサル（本番データ移行前に、テスト環境またはサンプルデータで新電子カルテへの取込を試行）

### 4. 移行運用・切替計画

- [ ] 切替日・移行作業のスケジュール（ダウンタイム有無、休診日実施の要否）を決定
- [ ] 移行中〜移行後の並行稼働期間の有無と運用ルール（両カルテへの二重入力が発生しないか）
- [ ] 移行失敗時のロールバック手順（EyeData側DBのバックアップ取得、Inno側を復旧可能な状態に維持する期間）
- [ ] 移行完了後のInnoKarte関連設定・依存コードの後片付け（不要になった`InnoButton`・DBリンク・`InnoUketsukeLib`参照等の削除判断）
- [ ] スタッフへの操作変更周知（現行カルテ起動ボタンの挙動変更、ログイン方式変更など）

### 5. テスト・検証

- [ ] `EyeData.Tests`の既存テスト（`FormExportTests`等）が改修後も通ることを確認
- [ ] エクスポート→新電子カルテ取込の一連の流れをステージング環境で通しテスト
- [ ] 移行後、EyeData起動〜ログイン〜患者検索〜各種登録機能が新電子カルテ連携済み設定で正常動作することを確認

## 未確定・要質問事項（再掲）

- 患者IDが新電子カルテでも維持される前提で問題ないか（変わる場合はID変換作業が追加で発生）
- 現行の「カルテ起動ボタン」に相当する機能を新電子カルテ側にも用意するか（無い場合、EyeDataからのカルテ参照導線をどうするか）
