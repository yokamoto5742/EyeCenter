# 本番環境 Oracle.DataAccess 読み込みエラー対応 - 注意点

## 対応内容
- `Program.cs` に `AssemblyResolve` フォールバックを追加し、config のバージョンリダイレクトが
  パッチバージョン不一致で失敗した場合、exe と同フォルダの `Oracle.DataAccess.dll` を
  バージョン不問で読み込むようにした。
- `EyeData.exe.config.production` の冒頭コメントを実態に合わせて更新。

## デプロイ時の注意点
- **新しい `EyeData.exe` を本番機に配置し直すこと。** `Program.cs` の修正なので、DLL や
  config だけ差し替えても効果はない。
- 本番機の `Oracle.DataAccess.dll`（2.112系）はそのままで良い。今回のフォールバックは
  「配置済みDLLをバージョン不問で読み込む」もeのなので、DLL自体の削除・置き換え作業は不要になる。
  ただし config の `bindingRedirect` 自体は残しているため、**削除する運用も引き続き有効**。
- `deploy-production.ps1` の除外リスト（`Oracle.DataAccess.dll` を配布物から除外する処理）は
  変更していない。本番機に既存の DLL が無い状態で新規配置する場合は、
  別途 Oracle 11.2 クライアントの DLL を手動配置する必要がある（従来通り）。

## 動作確認できていない点
- 本番機の実機でのアセンブリ解決（フォールバック経路）は未検証。x86 Debug でのビルド成功のみ確認済み。
- 本番機にログインして起動確認するまでは、今回の修正で実際にエラーが解消するかは未確定。

## 副作用・リスク
- `AssemblyResolve` イベントは `Oracle.DataAccess` という名前のアセンブリ要求にのみ反応するよう
  フィルタしているため、他のアセンブリ解決には影響しない。
- 開発機（config が 4.122→4.122 の一致リダイレクト）では通常通り解決されるため、
  フォールバックは発動しない＝開発機の挙動に変更なし。

## 今後の切り分けに使えるポイント
- 同種の「タイプ初期化子が例外をスローしました」系エラーが今後出た場合、
  まず `[InnerException]` の2段構造（要求バージョン→リダイレクト先バージョン）を見れば
  「厳密名アセンブリのバージョン不一致」かどうかを即判定できる。
