# 電子カルテ切替に伴うデータ移行プラン

対象: EyeCenter（眼科業務アプリ）が現行Oracle DBに保持するデータを、CSVエクスポート経由で新電子カルテ（DBはOracle）へ全件移行する。

作成日: 2026-07-04

---

## 1. 前提と現状把握

### 1.1 現行のCSVエクスポート機能（`FormExport.cs`）

エクスポートはメイン画面のエクスポートボタンから起動する（**利用可能ユーザーはID 519 / 363 / 305 / 752 / 1034 に限定**。`MainForm.cs:172`）。3種類のデータを出力できる。

| データ | 取得元テーブル | 出力仕様 |
|---|---|---|
| 検査データ | `EYE_KENSA` | ヘッダー1行。列: PATIENT_ID, KENSA_ID, KENSA_NAME, KENSA_DATE, CONT, STAFF, SAVE_DATE, SAVE_TIME, PDF_SAVE |
| 手術記録 | `EYE_OPE_RECORD` + `EYE_OPE`（IDで結合） | ヘッダー2行（1行目=コード、2行目=名称）。CONT列を `OpeTabItem` マスタ定義の項目別カラムに展開して出力 |
| サマリー | `EYE_SUMMARY`（患者氏名は `PatBase` で補完） | ヘッダー2行。CONT1〜CONT4を `SumItem1〜4` マスタ定義の項目別カラムに展開して出力 |
| 患者【追加済み】 | 患者マスタ（`PatBase` 経由） | ヘッダー1行。EYE系全テーブルに登場する PATIENT_ID を対象に、PATIENT_ID, NAME, KANA, SEX, BIRTH, TEL, POST, ADDR1, ADDR2, DEAD を出力。患者マスタに存在しないIDは空欄で出力（孤児レコード検出用） |
| 手術予約【追加済み】 | `EYE_OPE` | ヘッダー1行。全カラム＋OPE_KIND_NAME（種別名称）を全件出力（取消済み STATUS=0 を含む）。手術記録エクスポートに含まれない記録未作成の予約もカバー |

共通仕様:

- **文字コード: Shift-JIS（既定）／UTF-8（BOM付き）を画面で選択可能**
- 全セルをダブルクォートで囲み、`"` は `""` にエスケープ
- 主キー順のページング方式（5,000件/ページ）でメモリを使わず全件をストリーム出力
- 検査データの CONT 内改行は `<CR+LF>` トークンに置換され、**1レコード=1物理行**（手術・サマリーは項目展開時に `<CR+LF>` を実改行に戻すため、セル内改行が発生し得る）
- 完了時に総件数をダイアログ表示し、件数照合用マニフェスト（対象・ファイル名・件数・出力日時）を `<CSVファイル名>.manifest.txt` として同じ場所に出力

### 1.2 スコープに関する決定事項（2026-07-04 確定）

| データ | 決定 |
|---|---|
| **患者マスタ**（氏名・生年月日等） | 現行カルテベンダーの移行ツールで移行する。バックアップとして EyeCenter に患者CSVエクスポート機能を追加済み（§1.1） |
| **手術予約**（EYE_OPE） | **移行対象に含める**。手術記録エクスポートは EYE_OPE_RECORD との結合のため記録未作成の予約を含まないことが判明したため、EYE_OPE 全件のエクスポート機能を追加済み（§1.1） |
| **PDF・画像等のファイル資産** | **今回スコープ外**。CSVには出力しない（常時空だったPDF関連カラムもCSVから削除済み） |

### 1.3 データ規模

- 検査データ: 数十万件（最大ボリューム。エクスポート時間・ファイルサイズの実測が必須）
- 手術記録・サマリー: 検査より小規模

## 2. 移行スケジュール（数日間のダウンタイム活用）

### ステップ1: 事前準備・クレンジング（移行1ヶ月前〜直前）

1. **コード変更の実施とリリース**（§6）。特に文字コードオプション・件数マニフェスト出力は模擬エクスポート前に入れておくと検証が一度で済む。
2. **テスト環境での模擬エクスポート**
   - 手術・サマリー・検査・患者・手術予約の全件エクスポートを実施
   - 出力時間・ファイルサイズを計測し、当日のタイムテーブルを確定
   - ローカル検証環境（FREEPDB1のテストDB）でも手順のリハーサルが可能
3. **CSVサンプルと仕様書を新ベンダーへ提供**
   - 列定義（手術・サマリーはマスタ定義 `OpeTabItem` / `SumItem1〜4` により列が動的に決まるため、**本番マスタで出力したヘッダーをそのまま仕様として提示する**）
   - `<CR+LF>` トークン、`""` エスケープ、ヘッダー行数、文字コード、日付項目の書式を明記
4. **データクレンジング**: 模擬エクスポート結果から不正データ（欠損PATIENT_ID、日付異常等）を洗い出し、移行前に修正
5. **新ベンダーとの取込テスト**: サンプルCSVで新DB（またはステージングテーブル）への取込を1回以上通す

### ステップ2: 現行システムの停止と最終抽出（移行第1日目）

1. 現行システムへの入力受付を完全停止（読み取り専用化）。EyeCenterの入力系画面の利用を停止し、全端末で保存操作が発生しないことを確認
2. 最終データ抽出:
   - 手術記録CSVエクスポート
   - サマリーCSVエクスポート
   - 検査データCSVエクスポート（数十万件・最長時間。模擬エクスポートの実測時間を確保）
   - 手術予約CSVエクスポート（切替日以降の予約を新システムへ登録するため）
   - 患者CSVエクスポート（ベンダー移行のバックアップ・突合用）
3. 各ファイルの**件数・サイズ・ハッシュを記録**（§6.5のマニフェストを利用）し、媒体2系統（例: 移行用サーバー＋外付けディスク）にバックアップ

### ステップ3: データ変換と新DBへの取込（移行第2〜3日目）

1. 新カルテの受け入れ仕様に合わせてレイアウト・文字コードを変換（新ベンダーと共同作業）
   - 患者ID体系が変わる場合はID変換テーブルを介してPATIENT_IDを付け替え
   - `<CR+LF>` トークン → 新システムの改行表現への変換
   - 日付・時刻項目（KENSA_DATE / OPE_DATE / SAVE_DATE / SAVE_TIME 等）の書式変換
2. 新Oracle DBへ直接ロード（SQL*Loader / ステージングテーブル）またはインポートAPI経由で取込
3. 取込エラーレコードは全件リスト化し、原因判別（データ不良／仕様齟齬）→ 修正 → 再取込

### ステップ4: 移行検証・リハーサル（移行最終日）

1. **件数照合**: エクスポート時マニフェストの件数 = 新DBの各テーブル件数 を突合
2. **サンプル患者の画面突合**: 手術・検査履歴が豊富な患者を事前に5〜10名指定し、新システム画面と現行EyeCenter画面（読み取り専用で温存）を目視突合
   - 日本語文字列（患者氏名・所見テキスト）の文字化け有無を重点確認
   - セル内改行が新システムで正しく表示されるか確認
3. 検証合格後、新システム稼働開始
4. 現行DB・エクスポートCSVは保存年限まで読み取り専用で保全

## 3. 新ベンダーへの確認事項（早期確定が必須）

1. **標準CSVインポート機能の有無とフォーマット仕様**
   - 患者・手術・検査・サマリーをそれぞれCSVで一括取込できる口があるか
   - ある場合、ヘッダー定義・データ型（日付フォーマット等）の仕様書の提供を要求
   - **セル内改行・改行トークンの受け入れ可否**（検査CONTの `<CR+LF>`、手術・サマリーの実改行）
2. **Oracle DBへの直接書き込み（ステージングテーブル提供）の可否**
   - APIがない場合、ベンダー提供のステージングテーブルへCSVを流し込み、ベンダー側スクリプトで本番テーブルへコンバートしてもらえるか
3. **外字・文字コードの扱い**
   - 現行はShift-JIS。新システム（通常UTF-8）取込時の文字化け対策・変換フィルターの有無
   - Shift-JISに存在しUTF-8変換で問題になりやすい文字（外字領域、機種依存文字）の検出・変換ルール
4. **患者ID体系**: 現行PATIENT_IDをそのまま使えるか、ID変換が必要か
5. **手術記録・サマリーの可変列**: マスタ定義で列が決まる動的レイアウトを受け入れられるか、固定レイアウトへの変換が必要か

---

## 4. リスクと対策

| リスク | 対策 |
|---|---|
| Shift-JIS→UTF-8変換での文字化け（外字・機種依存文字） | ステップ1の模擬エクスポートで全ファイルをUTF-8変換にかけ、変換不能文字を全件リスト化して事前に対処方針を決める |
| 検査データ（数十万件）のエクスポート・取込が想定時間を超過 | ステップ1で実測し、タイムテーブルに1.5〜2倍のバッファを確保 |
| エクスポート中のクライアントPC障害・画面クローズ（`IsDisposed`で中断される仕様） | 実行端末を固定し、当日は移行作業専用にする。完了ダイアログの件数を必ず記録 |
| 移行後に現行データへの参照が必要になる | 現行DB・EyeCenterを読み取り専用で一定期間温存 |
| セル内改行・トークンの解釈齟齬で取込データが崩れる | サンプルCSVでの取込テスト（ステップ1-5）を必須とし、改行を含むレコードを意図的にサンプルへ含める |

## 6. コード変更（2026-07-04 実装済み）

対象は `FormExport.cs`・`FormExport.Designer.cs`。6.3以外は実装済み。

### 6.1 UTF-8（BOM付き）出力オプションの追加【実装済み】

エクスポート画面に「UTF-8（BOM付き）で出力」チェックボックスを追加。既定は従来どおりShift-JIS。マニフェストもCSVと同じ文字コードで出力される。BOM付きにしたのはExcel等での検証時に文字化けしないため。

### 6.2 `WriteCsvLine` の行末カンマ除去【実装済み】

手術・サマリーCSVの全行末に付いていた余分な空カラム（末尾カンマ）を除去し、検査データと同じ `string.Join(",", ...)` 方式に統一した。**旧形式のCSVを前提にした取込設定がある場合は列数が1減る点に注意。**

### 6.3 手術・サマリーのヘッダー1行化オプション【未実装・仕様確定待ち】

手術・サマリーはヘッダーが2行（コード行＋名称行）のまま。サマリーには同一コードが2列並ぶ項目（SumItem2の2値項目）があり、単純な1行化では列名が重複するため、新ベンダーの取込仕様が確定した時点で列名の付け方を決めて対応する。

### 6.4 常時空のPDF関連カラムの削除【実装済み】

手術記録CSVの `PDF_SAVE` / `PDF_DATE` / `PDF_TIME`、検査CSVの `PDF_SAVE` を削除した（PDF資産はスコープ外決定のため）。

### 6.5 件数マニフェストの出力【実装済み】

全エクスポートの完了時に `<CSVファイル名>.manifest.txt`（対象・ファイル名・件数・出力日時）を出力する。ステップ4の件数照合のエビデンスとして使用する。

### 6.6 患者CSVエクスポート機能の追加【実装済み】

「患者（患者マスタ）」ラジオボタンを追加。EYE_KENSA・EYE_OPE・EYE_SUMMARY に登場する全 PATIENT_ID を対象に、患者マスタから PATIENT_ID, NAME, KANA, SEX, BIRTH, TEL, POST, ADDR1, ADDR2, DEAD を出力する。患者マスタに存在しないIDは空欄行として出力されるため、孤児レコードの検出にも使える。位置づけは現行カルテベンダー移行の**バックアップ**。

### 6.7 サマリー `SumItem2` の空白分割の修正【実装済み】

半角スペースでの分割を最大2分割（`Split(new char[] { ' ' }, 2)`）に変更し、値に空白が2つ以上含まれる場合の3トークン目以降の欠落を解消した。

### 6.8 手術予約CSVエクスポート機能の追加【実装済み】

「手術予約（EYE_OPE）」ラジオボタンを追加。EYE_OPE の全カラム（予約・入院・感染症・同意等の全項目＋DEL系カラム）に OPE_KIND_NAME（種別名称）を加えて全件出力する。取消済み（STATUS=0）も含むため、新システム側で STATUS により取捨選択する。

## 7. 実施ステップまとめ（検証ポイント付き）

```
1. 新ベンダーへ§3を照会・仕様確定       → 検証: 取込仕様書とサンプル取込結果の受領
2. コード変更（§6.1〜6.5）             → 検証: テスト環境で3種エクスポートし列数・文字コード・件数を確認
3. 模擬エクスポート＋時間計測           → 検証: 全件出力成功、件数がDB件数と一致、変換不能文字ゼロ
4. 新ベンダーと取込テスト               → 検証: サンプル患者の画面突合OK
5. 本番移行（ステップ2〜4）             → 検証: 件数照合＋サンプル患者突合の合格
```
